### 2026-06-20 — FIX build de prod (NO era serwist) — root cause = `@swc/helpers` lockfile drift ✅ — `npm ci` + build verdes
La "deuda de serwist" documentada antes era un **diagnóstico incorrecto**. Producción (Vercel/Linux) siempre compiló bien (sw.js de 50KB live). El build local (Windows) y **CI** fallaban por otra cosa:
- **CI fallaba en `npm ci`** (no en el build): `npm error Missing: @swc/helpers@0.5.23 from lock file`. Causa: `@swc/core@1.15.41` (vía `next-intl@4.13.0`) tiene **peerDependency `@swc/helpers>=0.5.17`**, pero `next@15.5.19` fija exacto `0.5.15`. El lock committeado quedó inconsistente (npm 11 local lo toleraba; npm 10 de CI lo rechaza). Borrar node_modules+lock no bastó (npm 11 re-dedupea 0.5.15).
- **El build local de serwist** fallaba con `webpack-runtime "Cannot read properties of undefined (reading 'call')"` en `/_error` — **mismo root cause**: los helpers de SWC se inyectan en cada módulo transpilado; el `@swc/helpers@0.5.15` inválido rompía el runtime. Serwist solo lo *exponía* (su pass extra de webpack tocaba `/_error`).
- **Fix**: `overrides: { "@swc/helpers": "0.5.23" }` en `package.json` (satisface a ambos; los helpers 0.5.x son API-compatibles) + lock regenerado. Verificado: `npm ci` **exit 0** · `npm run build` **exit 0 (30/30 páginas, serwist HABILITADO)**.
